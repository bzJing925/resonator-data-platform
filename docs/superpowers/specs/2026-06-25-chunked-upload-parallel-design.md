# 上传流水线解压-计算并行化设计

## 背景与目标

当前 `aln.extract_batch` → `aln.compute_batch` 是串行 Celery chain，必须等整个 zip 解压（以及全部去嵌）完成后才开始指标计算。对于 `/Users/jingbozuo/Projects/#2.zip` 这类 ~10 GB / 2 万 + 文件的 Deflate64 归档，解压耗时不可忽略，计算资源在解压阶段大量空闲。

本设计目标：在 `feat/backend-api` worktree 中实现**块级并行**，让 DUT 文件按 chunk 分批解压并计算，chunk N+1 解压时 chunk N 可以同时做指标计算。

## 关键决策

| 决策项 | 选择 | 理由 |
|---|---|---|
| 并行粒度 | 块级（每块一批 DUT 文件） | 保留 7z 对 Deflate64 的高效解压；单 chunk 失败范围可控；调度简单 |
| 校准件策略 | 先完整解压全部 OPEN/SHORT 校准件，再分块解压 DUT | 校准件数量通常 << DUT 数量；复用现有去嵌逻辑；避免为每个 chunk 维护校准件映射表 |
| 调度方式 | Celery `group` + `chord` | 项目已使用 Celery + Redis 作为 broker/result backend，chord 是标准模式 |
| chunk 大小 | 默认 500 个 DUT 文件，可配置 | 兼顾并行度与单 chunk 开销；#2.zip 约可分成 40+ chunk |
| 进度表 | 新增 `batch_chunks` 表 | 支持 SSE 汇总进度、任务详情页展示、兜底轮询、失败排查 |
| 旧路径兼容 | 保留 | 现有测试/EAGER 模式仍可走 `extract_batch` → `compute_batch` |

## 总体流程

```
upload ZIP
    │
    ▼
create_batch_and_dispatch()
    │
    ▼
_dispatch_parallel_chain()
    │
    ├── 若 deembed=true ──▶ extract_calibrations_task
    │                          解压全部 OPEN/SHORT .s2p
    │
    ▼
dispatch_chunks_task
    列出归档成员 → 分类 → 分块
    创建 batch_chunks 行
    返回 group((extract_chunk | compute_chunk) × N)
    │
    ▼
           ┌─────────────────────────────────────────────┐
           │  group of chunk pipelines (parallel)        │
           │  extract_chunk_0 | compute_chunk_0          │
           │  extract_chunk_1 | compute_chunk_1          │
           │  ...                                        │
           └─────────────────────────────────────────────┘
                            │
                            ▼
                 chord callback
                            │
                            ▼
                 finalize_batch_task
                 汇总 → 刷新物化视图 → 标记成功/失败
```

## 新增 / 修改模块

### 1. 数据模型

新增表 `batch_chunks`：

```sql
CREATE TABLE batch_chunks (
    id bigint generated by default as identity primary key,
    batch_id bigint not null references batches(id) on delete cascade,
    chunk_index int not null,
    status text not null default 'pending',
    device_count int not null default 0,
    failure_count int not null default 0,
    error_msg text,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique(batch_id, chunk_index)
);

CREATE INDEX idx_batch_chunks_batch_status ON batch_chunks(batch_id, status);
```

无需修改 `upload_tasks`、`batches`、`devices` 表结构；`devices` 已有 `(batch_id, s_param_path, s_param_port)` 唯一约束，用于 chunk 重试幂等。

### 2. 新增 Celery 任务

文件：`backend/app/workers/chunked_upload.py`

#### `extract_calibrations_task`

输入：

```python
{
    "upload_task_id": int,
    "zip_path": str,
    "batch_no": str,
    "deembed_enabled": bool,
}
```

行为：

1. 列出 zip 中所有 `.s2p` 成员。
2. 识别 OPEN/SHORT 校准件，用 7z 一次性提取到 `files/<batch_no>/`。
3. 返回校准件相对路径列表与 DUT 总数估算。
4. 更新 `upload_tasks` progress 0% → 10%。

#### `dispatch_chunks_task`

输入：

```python
{
    "upload_task_id": int,
    "zip_path": str,
    "batch_no": str,
    "batch_id": int,
    "mapping_id": int,
    "deembed_enabled": bool,
    "deembed_method": str,
    "f_start_ghz": float | None,
    "f_end_ghz": float | None,
    "process_type": str,
    "calibration_files": list[str] | None,  # 仅去嵌时需要
}
```

行为：

1. 列出 zip 中所有成员，分类出 DUT `.s2p`/`.s1p`。
2. 按 `chunk_size`（默认 500 DUT 文件）分块。
3. 在 `batch_chunks` 表中为每个 chunk 创建一行（`status='pending'`）。
4. 构造并返回 Celery `group`：

```python
header = group(
    chain(
        extract_chunk_task.s(...),
        compute_chunk_task.s(...),
    )
    for chunk in chunks
)
return header
```

> 注意：`dispatch_chunks_task` 本身由 Celery chord 调用，它的返回值作为 chord 的 `header`。

#### `extract_chunk_task`

输入：

```python
{
    "upload_task_id": int,
    "batch_id": int,
    "batch_no": str,
    "mapping_id": int,
    "chunk_index": int,
    "dut_relpaths": list[str],
    "deembed_enabled": bool,
    "deembed_method": str,
    "f_start_ghz": float | None,
    "f_end_ghz": float | None,
    "wafer": int | None,
}
```

行为：

1. 更新 `batch_chunks[chunk_index].status = 'running'`、`upload_tasks.stage = 'extract'`。
2. 用 7z `-i@list.txt` 仅提取该 chunk 的 DUT 文件到 `files/<batch_no>/`。
3. 若去嵌：
   - 扫描 chunk 内 DUT `.s2p`。
   - 复用已存在的校准件路径。
   - 调用 `_run_deembed_for_batch(...)` 生成 `*_de.s1p`。
4. 构造 `all_files`（结构与现有 `extract_batch_task` 输出兼容）。
5. 返回包含 `all_files` 的字典。

#### `compute_chunk_task`

输入：`extract_chunk_task` 的输出。

行为：

1. 更新 `batch_chunks.status = 'running'`、`upload_tasks.stage = 'metrics'`。
2. 与现有 `compute_batch_task` 相同，但只处理一个 chunk。
3. 删除该 chunk 对应 `(batch_id, s_param_path, s_param_port)` 的已有 device 行，再插入，保证幂等。
4. 更新 `batch_chunks.device_count / failure_count / status`。
5. 调用新的 `ProgressPublisher.aggregate_progress()`，根据所有 chunk 汇总 `upload_tasks.progress_pct`。
6. 返回 `{"batch_id": int, "chunk_index": int, "device_count": int, "failures": int}`。

#### `finalize_batch_task`

签名：

```python
@app.task(bind=True, name="aln.finalize_batch")
def finalize_batch_task(self, chunk_results: list, *, upload_task_id: int, batch_id: int) -> dict:
```

行为：

1. 检查所有 chunk 结果：若有任何失败，标记 `upload_tasks` failed，并聚合错误信息。
2. 若全部成功：
   - 汇总 `batch_chunks.device_count` → `batches.device_count`
   - `REFRESH MATERIALIZED VIEW CONCURRENTLY mv_batch_stats`
   - 按 `KEEP_RAW_ZIP` 清理原 zip
   - 标记 `upload_tasks.status = 'success'`
3. 返回 `{"batch_id": batch_id, "device_count": int, "failures": int}`。

### 3. 调度入口改造

文件：`backend/app/services/upload_service.py`

新增 `_dispatch_parallel_chain()`：

```python
def _dispatch_parallel_chain(
    task: UploadTask,
    zip_path: Path,
    batch_no: str,
    batch_id: int,
    mapping_id: int,
    *,
    f_start_ghz: float | None = None,
    f_end_ghz: float | None = None,
    deembed: bool = False,
    deembed_method: str = "default",
    process_type: str = "AUTO",
) -> str | None:
    from celery import chain, chord, group

    # 公共参数
    common = {
        "upload_task_id": task.id,
        "zip_path": str(zip_path),
        "batch_no": batch_no,
        "batch_id": batch_id,
        "mapping_id": mapping_id,
        "deembed_enabled": deembed,
        "deembed_method": deembed_method,
        "f_start_ghz": f_start_ghz,
        "f_end_ghz": f_end_ghz,
        "process_type": process_type,
    }

    # 去嵌时先提取校准件
    if deembed:
        pipeline = chain(
            extract_calibrations_task.s(**common),
            dispatch_chunks_task.s(**common),
        )
    else:
        pipeline = dispatch_chunks_task.s(**common)

    # dispatch_chunks_task 返回一个 group，chord 等待全部完成后回调 finalize
    result = chord(pipeline)(finalize_batch_task.s(upload_task_id=task.id, batch_id=batch_id))
    return result.id
```

`create_batch_and_dispatch()` 中增加一个切换开关（默认走并行路径）：

```python
USE_PARALLEL_UPLOAD = True  # 可改为从 Settings/环境变量读取

if USE_PARALLEL_UPLOAD:
    celery_task_id = _dispatch_parallel_chain(...)
else:
    celery_task_id = _dispatch_chain(...)
```

### 4. 归档分块工具

文件：`backend/app/workers/chunking.py`

提供纯函数：

- `list_zip_members(zip_path: Path) -> list[ZipMemberInfo]`：优先 `7z l -slt`，fallback `zipfile`。
- `classify_members(members) -> tuple[list[str], list[str]]`：返回 `(calibration_relpaths, dut_relpaths)`。
- `chunk_dut_files(dut_relpaths: list[str], chunk_size: int) -> list[list[str]]`：简单按顺序切分；可选按 mark/coord 聚类以减少校准件缓存切换。

### 5. 进度发布增强

文件：`backend/app/workers/progress.py`

新增 `ProgressPublisher.aggregate_progress(db, batch_id)`：

```python
def aggregate_progress(self, db: Session, batch_id: int) -> None:
    stmt = select(
        func.count(BatchChunk.id),
        func.sum(BatchChunk.device_count),
        func.sum(BatchChunk.failure_count),
    ).where(BatchChunk.batch_id == batch_id)
    total_chunks, device_count, failure_count = db.execute(stmt).one()
    # 根据已完成 chunk 比例 + 当前 chunk 阶段进度，更新 upload_tasks
```

SSE 接口 `/api/tasks/{id}/stream` 不需要改动，仍消费 `upload_tasks` 表。

## 数据流时序

```
T0: upload_task status=pending, progress=0
T1: extract_calibrations 开始 → progress 0~10
T2: dispatch_chunks 列出成员、分块、创建 batch_chunks → progress 10
T3: chunk 0..N 同时进入 extract_chunk → progress 10~30
T4: chunk 0 完成 extract 进入 compute_chunk；chunk 1..N 仍在 extract → progress 30~95
T5: 所有 chunk compute 完成 → finalize_batch → progress 95~100 → success
```

## 错误处理

| 场景 | 行为 |
|---|---|
| 单个 chunk extract 失败 | 该 chain 中断，compute_chunk 不执行；chord 回调收到失败结果 |
| 单个 chunk compute 失败 | chord 回调收到失败结果 |
| finalize 检测到失败 | 标记整个 upload_task failed；`error_msg` 包含失败 chunk 索引与错误 |
| 用户重新上传同一 batch_no | `create_batch_and_dispatch` 仍拒绝 409；用户删除旧批次后可重试 |
| chunk 重试 | `compute_chunk` 幂等删除再插入；`extract_chunk` 会覆盖已提取文件 |

## 幂等性

- `extract_chunk`：7z 覆盖提取；去嵌输出 `_de.s1p` 文件名固定。
- `compute_chunk`：开头执行 `DELETE FROM devices WHERE batch_id = ? AND s_param_path IN (...)`，再批量插入。
- `finalize_batch`：只汇总 `batch_chunks`，不直接插入 device。

## 测试策略

1. **单元测试 `chunking.py`**：
   - 列出 #2.zip 成员数量与分类正确
   - 分块大小边界
   - 校准件识别正确

2. **集成测试（小 Deflate64 子集）**：
   - 上传 100 DUT / chunk_size=20 → 产生 5 个 chunk
   - 验证校准件先解压
   - 验证多个 chunk 同时处于 running（通过 `batch_chunks` 状态）
   - 验证最终 device_count = 200（每个 DUT 拆 S11/S22）
   - 验证无重复 device 行
   - 验证 SSE 进度连续上升

3. **失败测试**：
   - 构造缺少 SHORT 的 zip → 相关 chunk 失败，任务最终 failed，错误信息可读

## 与现有代码的兼容性

- `extract_batch_task` / `compute_batch_task` / `process_batch_task` 保持原样，作为串行回退与 EAGER 测试入口。
- `upload_service.create_batch_and_dispatch` 默认切到并行路径，但保留开关可回退。
- 前端无需修改；可选在 TaskDetail 增加 chunk 列表展示。
- 数据库通过 Alembic 新增 `batch_chunks` 表。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| chord 依赖 Redis result backend | 项目已使用 Redis backend；并在 `batch_chunks` 表中保存状态，可作为兜底 |
| 大量 chunk 导致 Celery group 过大 | chunk_size 可配置；默认 500 使 #2.zip 约 40+ chunk，可控 |
| 校准件提取失败导致所有 chunk 失败 | 提取校准件是串行第一步，失败即整体失败，错误清晰 |
| 并发 chunk 同时写 `upload_tasks` 进度产生竞态 | `ProgressPublisher` 每次更新基于 `batch_chunks` 聚合，而非累加；SQLAlchemy 事务保证 |

## 后续可扩展

- 文件级真流式：从 7z stdout 直接读取并计算，适合超大规模归档，但实现复杂。
- 失败 chunk 单独重试：在 TaskDetail 增加“重试失败 chunk”按钮。
- 动态 chunk 大小：根据单文件大小和 worker 内存自动调整。
